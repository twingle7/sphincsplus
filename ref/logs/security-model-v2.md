# 安全模型 v2（M12）

## 1. 目标
- 明确 v2 阶段的安全目标、非目标、攻击面与验收口径。
- 为 `Fischlin 盲签 + 完整 STARK 证明` 提供统一风险边界。

## 2. 安全目标
- 完整性：无有效 witness 时，攻击者不能构造通过 `ShowVerify` 的对象。
- 机密性：`Show_v2` 不泄露 `sigma_com`、`m`、`r`。
- 绑定性：`pi_F_v2` 绑定 `pk/com/public_ctx/statement_version`，防止跨上下文重放。
- 一致性：C/Rust 后端验证结果一致，不出现同输入分歧。
- 可审计性：关键失败原因可观测（格式错误、绑定错误、证明无效）。

## 3. 非目标（当前阶段）
- 不在本阶段给出形式化 UC 证明。
- 不在本阶段完成网络层匿名性系统设计（如混淆网络、流量对抗）。
- 不在本阶段给出最终生产级抗侧信道实现。

## 4. 攻击面与防护要点
- 格式篡改：通过 `magic/version/header_len/total_len` 严格校验。
- 绑定篡改：校验 `public_input_digest` 与 `ctx_binding`。
- 重放攻击：proof transcript 绑定 statement/version/domain。
- 后端分歧：引入跨后端一致性测试，阻断“同输入异结论”发布。
- 隐私退化：禁止在 proof payload 中携带明文签名及可链接标识。

## 5. 必须满足的红线
- 红线 1：`pi_F_v2` 含 `sigma_com` 明文，判定失败。
- 红线 2：同一 `(pk, com, public_ctx, pi_F_v2)` 在 C/Rust 给出不同结论，判定失败。
- 红线 3：展示对象可被 Issuer 稳定关联到具体盲签请求，判定失败。

## 6. 验收检查单
- 协议层：盲化/去盲化/展示流程与角色边界实现一致。
- 格式层：`pi_F_v2` 编解码严格执行，保留区规则明确。
- 证明层：篡改 `com/public_ctx/proof` 任一关键字段均拒绝。
- 兼容层：`v1` 仅兼容验证，`v2` 默认生成。
- 工程层：核心测试可复现，失败信息可定位。
