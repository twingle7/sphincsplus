    - 新增测试目标 `test/poseidon2_adapter`。
- 风险点/兼容性说明：

  - Trace 容量采用固定上限（`MAX_CALLS/MAX_LANES`），超限会计入 `dropped_calls/dropped_lanes`。

## 2026-04-15 第 10 轮：`下一步计划-v1.md` 可行性评估
- 评估对象：
  - `logs/下一步计划-v1.md`
- 总体结论：
  - v1 计划技术路线清晰，分层合理，整体**可行**；
  - 但工作量跨越“协议分层 -> witness 规范 -> AIR 子系统 -> 展示层替换 -> 论文交付”，必须按里程碑拆解推进，不能并行硬上；
  - 以当前仓库状态，推荐先完成 M5/M6 文档与接口冻结，再进入 M7~M10 的 STARK 工程实现。
- 分阶段可行性判定：
  - **M5（v1 边界冻结）**：高可行，几乎纯文档与接口语义冻结，依赖最少；可立即开始。
  - **M6（witness 格式冻结）**：高可行，依赖现有 `HashCall` 与 adapter，可在不引入 AIR 的情况下完成格式与导出器规范化。
  - **M7a（Permutation AIR）**：中等可行，需先选定 STARK 后端/域约束表达方式；建议先做最小 proof harness。
  - **M7b（Sponge AIR）**：中等可行，强依赖 M7a 的稳定列布局与过渡约束。
  - **M8（HashCall 序列 AIR）**：中等偏高风险，涉及调用拼接、长度语义、容量边界与 dropped 处理的一致性。
  - **M9a（最小 verify_com 语义）**：中等可行，建议先做“连接点正确性”而非全语义覆盖。
  - **M9b（完整 verify_com/SPHINCS+ 语义）**：高风险高工作量，需严格模块化推进，否则验证复杂度爆炸。
  - **M10（show 层替换）**：中等可行，前提是 M9 至少具备可用 `pi_F_v1`；否则只能保持占位对象。
  - **M11（测试/统计/论文）**：高可行，但依赖前述实现产物稳定后再收敛指标。
- 主要前置依赖与风险点：
  - 需尽早冻结 v1 外部对象：`Show_v1=(com, pi_F, public_ctx)`，并禁止复用 `spx_p2_bsig_public`。
  - `public_ctx/aux` 若不先冻结为空保留位，后续 witness/序列化/AIR 会反复返工。
  - STARK 后端选型（自研约束系统 or 现成框架）是关键前置；未定会阻塞 M7 起所有实现。
  - 当前 `HashCall` 已可导出，但 `addr_words` 解析依赖输入布局约定，布局变化需版本化处理。
  - 性能不是 v1 首要目标；在 M10 前不建议过早投入重优化。
- 建议执行顺序（与 v1 文档一致，附实现化建议）：
  - 先做 M5：产出 `stark-proof-spec-v1.md`，冻结可见性边界与对象分层。
  - 再做 M6：产出 `witness-format-v1.md` + `witness_builder` 最小实现与测试。
  - 然后 M7a/M7b：先 permutation、后 sponge，保证每层可单独 prove/verify。
  - 再推进 M8/M9a：将调用序列与最小验签语义闭合。
  - 最后 M9b/M10/M11：完整语义、展示层替换与论文材料。
- 与当前代码基线的一致性判断：
  - 当前 v0（公开 `sigma_com`）与 v1（隐藏 `sigma_com`）分层方向与 v1 文档一致；
  - 已有 adapter/trace 基础可复用，不需要推倒重来；
  - 需要新增 `show_poseidon2_v1.*` 与 `stark/*` 子树实现，属于增量演进而非重构式迁移。

## 2026-04-20 第 42 轮：修复 Rust STARK prover/verify 公共输入不一致导致 v2-strict 验证失败
- 改动文件：
  - `stark-rs/src/lib.rs`
  - `logs/开发日志.md`
- 问题现象：
  - 在启用 `SPX_P2_USE_RUST_STARK` 后，`v2-strict` 路径下：
    - `show_prove_v2_strict` 可以生成 `pi_F_v2`；
    - 但 `show_verify_v2_strict` 返回失败（`FAIL: show_verify_v2_strict` 等）。
- 根因分析：
  - Rust prover 侧：
    - 在 `WorkProver::get_pub_inputs` 中，把 `mix/bind` 固定为 `BaseElement::ZERO`；
    - 即证明的公开输入实际是 `(start, result, 0, 0)`。
  - Rust verifier 侧：
    - 解码 `pi_F_v2` 后，根据 `public_ctx` 和 digest 重新计算 `mix/bind` 非零值；
    - 再用 `(start, result, mix, bind)` 调用 Winterfell `verify`。
  - 结果：prover 和 verifier 所在的 STARK 语句不一致，导致验证必然失败。
- 修复实现：
  - 将 `mix/bind` 纳入 `WorkProver` 状态，并在 prove/verify 两侧保持一致：
    - `struct WorkProver { options, mix, bind }`；
    - `WorkProver::new(options, mix, bind)`；
    - `get_pub_inputs` 返回 `{ start, result, self.mix, self.bind }`；
    - `spx_p2_rust_generate_pi_f_v1` 中构造 `WorkProver::new(options_96bits(), mix, bind)`。
  - 验证侧保持不变，继续从 `pi_F_v2` 中恢复 `mix/bind` 并传入 Winterfell。
- 影响与说明：
  - 现在 Rust 后端证明与验证在 `PublicInputs` 上保持一致，`v2-strict` 路径可以验证自己生成的 proof。
  - 该修复不会改变 `pi_F_v2` 字节级格式，仅修复 prover/verify 语句不一致问题。
