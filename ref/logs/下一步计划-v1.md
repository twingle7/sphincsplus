# Fischlin-SPHINCS+-Poseidon2-STARK 下一步计划 v1

## 文档目的

**文档名**：`下一步计划v1.md`。  
本规范用于在既有 v0 工程原型基础上，明确 v1 正式证明版的协议对象、接口分层、witness 边界、STARK 落地步骤与里程碑安排。  
本版本特别解决一个核心冲突：v0 采用“公开 `sigma_com` 并直接验签”的原型口径，而 v1 采用“隐藏 `sigma_com`、仅对外展示零知识证明”的正式展示口径，因此二者不得继续混用同一层展示 API。

本规范的目标不是继续扩展 v0 的公开签名对象，而是把 v0 冻结为过渡原型，并在此基础上推进 v1 的匿名展示式证明系统实现。  
下一阶段的首要目标是：**将占位 `pi_F` 替换为真正的 STARK 证明，并使外部验证者在不直接看到 `sigma_com` 的情况下完成验证。**

---

## 0. 当前基线

当前工程已具备如下基础：

- 已完成 Poseidon2 后端接入，并在 `ref + sphincs-poseidon2-192s + simple` 路径下接管底层哈希调用。
- 已完成 `commit / verify_com / trace_verify_com / encode_bytes_to_lanes` 四个稳定接口。
- 已冻结 `HashCall` trace 记录规范。
- 已完成 v0 端到端原型：`Commit -> Issue(Sign com) -> ProveKnowledge(trace) -> Verify`。
- v0 中 `pi_F` 仍为占位承诺，不是真正的 Fischlin / STARK 证明对象。

因此，下一阶段不再以“扩展 v0 公开签名对象”为目标，而是以“构造隐藏 `sigma_com` 的正式展示版本”为目标。

## 1. v0 与 v1 的关系

### 1.1 版本分层说明

v0 与 v1 不是同一个接口的小改版，而是**两个不同阶段的协议对象模型**。

- **v0**：公开 `sigma_com` 的过渡原型；
- **v1**：隐藏 `sigma_com` 的正式展示证明版；
- 两者共享底层语义接口，但公开对象与验证方式不同。

若不显式区分这一点，读者会误以为只是在 `spx_p2_bsig_public` 上做小修小补；这与现有 v0 代码事实不符，因为 v0 的 `spx_p2_bsig_public` 明确包含 `sigma_com`，且 `spx_p2_bsig_verify` 直接依赖它完成公开验签。

### 1.2 v0 的定位

v0 是**过渡原型**，其用途是：

- 调通 `commit -> issue -> prove -> verify` 的整体骨架；
- 验证 adapter、trace 导出与占位 `pi_F` 接口的连通性；
- 为后续 witness 生成、AIR 设计和 STARK 证明提供工程底座。

v0 的验证模型是“公开签名对象验证”：

- `spx_p2_bsig_public` 显式包含 `com`、`sigma_com`、`pi_f`；
- `spx_p2_bsig_verify` 直接基于公开的 `sigma_com` 调用 `spx_p2_verify_com(pk, com, sigma_com)`；
- v0 中的 `pi_f` 由 `com || sigma_com || trace_meta` 导出，因此其绑定方式本身依赖公开保存的 `sigma_com`。

因此，**v0 不能直接当作匿名展示协议**；它只能作为“公开签名原型”的实现基线。

### 1.3 v1 的定位

v1 是**正式证明版**，其用途是：

- 隐藏 `sigma_com`；
- 只对外展示 `com` 与零知识证明对象；
- 让验证者验证“我知道一个对 `com` 的有效签名”，而不是直接看到该签名本体。

因此，v1 的验证模型不再是“公开签名对象验证”，而是“隐藏签名本体的证明式展示验证”。

### 1.4 共享部分与差异点

v0 与 v1 共享如下底层语义与实现资产：

- `spx_p2_commit`
- `spx_p2_verify_com`
- `spx_p2_trace_verify_com`
- `spx_p2_encode_bytes_to_lanes`
- `HashCall` 记录语义
- Poseidon2 编码与 sponge 规则

但 v0 与 v1 的**外部展示对象**必须严格区分：

- v0：公开 `sigma_com` 的过渡原型；
- v1：隐藏 `sigma_com` 的正式展示版。

## 2. 协议对象与可见性边界

### 2.1 v1 目标语句

v1 阶段唯一合法的证明语句定义为：

存在私有见证 `(m, r, sigma_com, trace, aux)`，使得：

1. `com = Commit(m || r)`；
2. `Verify(pk, com, sigma_com) = 1`；
3. `trace` 是执行 `spx_p2_trace_verify_com(trace, pk, com, sigma_com)` 所导出的合法轨迹；
4. `trace` 中所有 `HashCall`、地址字、输入长度、lane 编码、padding、输出长度均满足既有 trace 规范与 Poseidon2 编码规则；
5. 对外验证时无需直接公开 `sigma_com`。

### 2.2 公开输入与私有见证

v1 阶段的公开输入统一定义为：

- `pk`
- `com`
- `public_ctx`

v1 阶段的私有见证统一定义为：

- `m`
- `r`
- `sigma_com`
- `trace`
- `aux`

在 v1 中，`sigma_com` **必须作为私有 witness 处理**，不得继续出现在公开展示对象里。  
若未来需要单独研究“公开签名版展示”，必须另立版本，不得与 v1 混写。

### 2.3 `public_ctx` 与 `aux` 的当前约定

为避免后续 witness 格式和序列化规则反复返工，v1 最小实现阶段对 `public_ctx` 与 `aux` 统一作如下约定：

- `public_ctx`：在 **v1 最小实现** 中固定为空字节串，序列化时记为长度为 0 的可选公开字段；
- `aux`：在 **v1 最小实现** 中固定为空内部辅助字段，不进入公开输入，不单独参与外部对象序列化；
- 若未来扩展上下文绑定、展示域分离、应用标签或额外 witness 辅助量，必须通过版本化方式单独升级，不得直接修改 v1 最小实现口径。

这意味着在 M5 之前，`public_ctx` 与 `aux` 不是开放语义字段，而是**冻结为空的保留位**。

### 2.4 v1 对外展示对象

v1 阶段的对外展示对象统一定义为：

`Show_v1 = (com, pi_F, public_ctx)`

其中：

- `com` 为公开承诺；
- `pi_F` 为正式 STARK 证明对象；
- `public_ctx` 在 v1 最小实现中固定为空，但保留序列化位置；
- `sigma_com` 不出现在外部展示对象中。

### 2.5 内部凭证材料

为避免 v0 与 v1 语义混淆，v1 中应显式区分“持有者私有状态”和“外部展示对象”。  
建议新增一类内部凭证材料结构：

`Cred_v1_internal = (m, r, com, sigma_com, trace, aux)`

该结构仅供 prover / holder 本地使用，不对外公开。

## 3. 接口分层与命名统一

### 3.1 底层 helper 保留

以下接口在 v1 中继续保留，但其定位调整为 **prover 内部 helper**，而不是最终展示层 API：

- `spx_p2_verify_com(pk, com, sigma_com)`
- `spx_p2_trace_verify_com(trace, pk, com, sigma_com)`

它们的职责是：

- 复用既有 SPHINCS+ 验签逻辑；
- 为 witness 生成与 trace 导出服务；
- 为 STARK prover 构造私有执行轨迹提供一致的执行基线。

最终外部验证者不应直接调用这两类接口来处理展示对象，因为外部验证者在 v1 中不应直接拿到 `sigma_com`。

### 3.2 上层 API 重新分层

建议将 v1 的接口分为三层：

#### A. Issuer / Holder 内部层

- `issue(...)`：生成或交付内部凭证材料；
- `prepare_witness(...)`：从内部凭证材料生成 witness；
- `build_trace(...)`：从 `trace_verify_com` 导出规范化 trace；
- `show_prove(...)`：消费内部凭证材料并生成 `Show_v1`。

#### B. Presentation 层

- `show_prove(...) -> Show_v1`
- `show_verify(Show_v1)`

该层只处理外部展示对象，不再直接暴露 `sigma_com`。

#### C. 底层语义层

- `commit`
- `verify_com`
- `trace_verify_com`
- `encode_bytes_to_lanes`

这一层继续作为共享基础语义层，不直接承担对外展示职责。

### 3.3 命名统一规则

本版起，v1 **统一采用 `show` 术语体系**，不再在正式计划中并列保留 `bsig_poseidon2_v1` 与 `show_poseidon2_v1` 两套命名方向。  
原因是 v1 的对外对象已明确是 `Show_v1 = (com, pi_F, public_ctx)`，其语义重点是“展示证明对象”，而不是“公开签名对象”。

因此，v1 统一采用如下命名：

- `show_poseidon2_v1.h`
- `show_poseidon2_v1.c`
- `spx_p2_show_v1`
- `spx_p2_show_prove`
- `spx_p2_show_verify`

若后续仍需保留 `bsig` 一词，则仅将其视为**项目总称**，不再作为 v1 外部对象、文件名或测试名的主命名。

### 3.4 明确禁止复用的对象

**不得复用 `spx_p2_bsig_public` 作为 v1 的对外对象。**  
原因是该结构天然包含 `sigma_com` 字段，语义上属于 v0 的“公开签名对象”，不适合作为 v1 的外部展示对象。

## 4. 哈希、trace 与 witness 约束

### 4.1 Poseidon2 基线

哈希后端继续采用当前 Poseidon2 sponge 基线：

- 状态宽度 `t = 12`
- capacity = 6 lanes
- rate = 6 lanes
- 每次吸收/输出 48 字节

后续所有 witness 编码、AIR 约束、测试向量与 prover 实现均必须复用当前 Poseidon2 语义入口，不得重新发明第二套编码规则。

### 4.2 编码规则

- 所有输入先解释为单一 little-endian 字节流；
- 再按 8 字节分块装载为 64-bit lane；
- 不足 8 字节的尾块按零扩展；
- finalize padding 唯一固定为当前 sponge 实现中的 `0x01 ... 0x80` pad10*1 风格。

### 4.3 Trace 记录

所有哈希调用继续统一收敛为 `HashCall`。  
STARK 侧 witness 构造、行布局与 AIR 约束必须基于 `HashCall` 展开，而不是重新定义另一套“更抽象但不一致”的调用表示。

### 4.4 Witness 生成原则

- witness 生成器只能消费既有 trace 或与其严格同构的规范化表示；
- witness 生成器不得偷偷重写另一套验签逻辑；
- `verify_com` 与 `trace_verify_com` 必须保持同一路径；
- 任意展示证明若要成立，必须能追溯到一致的内部验签执行。

## 9. 本版修改结论

本版计划相较于旧版，完成了以下关键修正：

- 单独增加了“v0 与 v1 的关系”章节，不再将版本迁移说明散落在其他段落中；
- 明确冻结 v0，不再修改其语义，只将其定义为公开 `sigma_com` 的过渡原型；
- 明确 v1 为隐藏 `sigma_com` 的正式展示版；
- 将“内部凭证材料”和“外部展示对象”彻底拆层；
- 将 `verify_com / trace_verify_com` 重新定位为 prover 内部 helper；
- 明确禁止在 v1 中复用 `spx_p2_bsig_public` 作为外部对象；
- 明确 `public_ctx` 与 `aux` 在 v1 最小实现中固定为空保留位；
- 统一采用 `show_poseidon2_v1.*` 与 `test/poseidon2_show_v1.c` 的命名体系；
- 将测试计划拆分为 v0 兼容测试与 v1 正式展示测试两套。

以上修改的核心目标是：**让 v0 与 v1 在协议对象、接口语义和论文叙事上彻底解耦，同时最大化复用已有底层工程资产。**
