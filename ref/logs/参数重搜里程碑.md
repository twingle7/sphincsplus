# SPHINCS+ Poseidon2 参数重搜里程碑（TODO）

## 1. 对 `参数重搜.md` 的合理性评估

### 1.1 总体判断
- 结论：总体**合理且专业**，主线与 SPHINCS+ 参数设计流程一致。
- 亮点：把“接口实例化定义 -> 搜索空间 -> 结构过滤 -> 安全筛选 -> 实测 Pareto -> 安全表述”串成闭环，避免只做 benchmark 或只做口头安全主张。
- 亮点：明确引入签名预算 `q` 与超预算退化分析，符合参数设计实践，不是只盯单次 sign/verify。
- 亮点：强调 simple 路径的安全语境，避免把“工程可运行”误写成“完全等价标准参数”。

### 1.2 需补强点（从“研究建议”到“可执行计划”）
- 引文可追溯性：`[cite:*]` 目前是占位索引，需补成可复核文献清单（规范版本、章节、论文条目）。
- 搜索空间边界未量化：`(n,h,d,k,a,w,q)` 给了变量但没给扫描范围、步长、剪枝顺序，执行成本不可控。
- 安全计算口径未落脚本：需明确“组合层下界”由哪个脚本/公式实现，输入输出格式要固定。
- 约束统计口径需统一：当前仓库已有 `trace_calls/witness_rows/proof_bytes`，但“按 F/H/T_l/PRF/H_msg 分层约束数”尚需固化统计接口。
- 里程碑验收需可机读：建议每个阶段输出结构化 `csv/json`，减少人工拣数据。

### 1.3 与当前仓库基线的适配性
- 适配良好：仓库已有 Poseidon2 API/KAT、benchmark、STARK stats 与回归脚本，可支撑参数重搜落地。
- 现有能力可复用：`test/benchmark` 可做 keygen/sign/verify，`test/poseidon2_stark_stats` 可采 `trace/witness/proof` 指标。
- 关键差距：尚缺“参数枚举 + 安全筛选 + Pareto 输出”的一体化脚本链路。

## 4. 风险与缓解
- 风险：搜索空间过大导致运行时间失控。
- 缓解：先做 coarse grid，再局部细化；保留中间缓存。
- 风险：安全评估公式实现与规范口径偏差。
- 缓解：先用少量已知参数做交叉校验，再全量扫描。
- 风险：约束统计口径前后不一致，无法比较候选。
- 缓解：先冻结字段定义，再跑批量采集。
- 风险：最终结论过度主张。
- 缓解：严格沿用“targeted + simple/ROM + 预算条件”措辞。
