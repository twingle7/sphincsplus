# SPHINCS+ Poseidon2 参数重搜指导文档

## 文档目标

本指导文档用于支持一个已经具备稳定接口实现的 SPHINCS+ Poseidon2 simple 版本，完成从“可运行原型”到“可论证参数集”的下一阶段工作。SPHINCS+ 的安全分析框架本身将高层构造与底层哈希实例化分离，因此在替换底层哈希后，重点不是重写整个方案证明，而是明确新的函数族实例化方式，并结合参数评估方法重新判断安全级别。[cite:58][cite:59]

本阶段建议将目标安全强度设为 128-bit targeted classical security，而不是一开始就直接主张 192-bit；公开的 SPHINCS+-SM3 替换型实现工作首先实现和测试的是 128-f 模式，而不是直接以 192-bit 作为首个落地目标。[cite:56]

## 研究目标

### 总目标

为 SPHINCS+ Poseidon2 simple 版本重新搜索一组面向 128-bit 目标安全强度的参数，使该参数集在以下四个维度之间达到可接受平衡：

- 组合层安全下界满足目标。
- Poseidon2 实例化安全假设与接口设计可被明确陈述。
- 软件实现运行稳定，签名、验证和密钥生成成本可测量。
- 面向零知识应用时，底层哈希调用次数与约束数具有实际优势。[cite:58][cite:59][cite:12]

### 子目标

1. 明确 Poseidon2 对 SPHINCS+ 六类接口的实例化方式：\(T_\ell\)、\(F\)、\(H\)、\(PRF\)、\(PRF_{msg}\)、\(H_{msg}\)。SPHINCS+ 规范要求实例化必须精确定义这些函数族，而不是只说明“底层哈希替换为某个哈希函数”。[cite:58]
2. 建立参数搜索空间 \((n,h,d,k,a,w,q)\)，并对每个候选自动计算长度、签名大小、调用次数与安全估计值。SPHINCS+ 规范和参数说明文档都将参数搜索视为系统化流程，而非手工试错。[cite:58][cite:57]
3. 使用 SPHINCS+ 的参数评估口径筛除不满足 128-bit 目标的候选。规范第 9 章和相关参数说明文档都将“给定参数集的安全级别”作为可计算对象。[cite:58][cite:57]
4. 对通过解析筛选的参数做实现测试，输出 Pareto 前沿候选，而不是只给出单一“拍脑袋参数”。参数说明文档强调应比较签名长度、速度和签名次数预算等多个维度。[cite:57]
5. 对最终入围参数给出谨慎、可发表的安全表述。对于 simple 版本，更合理的表述是“在 simple/ROM 语境下针对 128-bit classical security 设计并评估”，而不是未经补充分析就直接声称完全等同标准参数集安全性。[cite:58][cite:59]

## 工作边界

### 已有基础

当前已有一个稳定可用的 SPHINCS+ Poseidon2 simple 接口，并且在 sha2-192s 对应参数集下运行稳定。这个结果说明工程接口已经具备继续做参数重搜的条件，但不能直接推出“该实现已经达到同等级安全”。[cite:58][cite:59]

### 本阶段不要求的内容

- 不要求重新实现 robust 版本。SPHINCS+ 规范明确区分了 simple 与 robust 两类实例化；既然当前研究路线已经确定为 simple，本阶段可专注 simple 方案的参数与安全评估。[cite:58]
- 不要求从零开始证明完整 SPHINCS+ 安全性。高层构造的安全框架已由 SPHINCS+ 论文给出，当前重点是新底层实例化与参数的适配分析。[cite:59]
- 不要求一开始就把目标推到 192-bit。已有替换型工作优先展示了 128 档实现，说明先做 128-bit 是合理且稳妥的研究起点。[cite:56]

## 参数重搜任务

### 第一步：固定实例化规范

需要先写出一份“接口实例化说明”，至少包括以下内容：

- Poseidon2 使用的有限域。
- 状态宽度、轮数、S-box 指数。
- 各接口输入如何编码到海绵或置换输入。
- ADRS 如何编码与域分离。
- 输出截断方式。
- 不同接口之间如何避免可混淆输入空间。

这一部分是参数重搜的前置条件，因为在 SPHINCS+ 中参与安全分析的对象并不是单一哈希函数名，而是具体实现后的函数族接口。[cite:58][cite:59][cite:12]

### 第二步：定义搜索空间

建议搜索以下参数：

- \(n\)：安全参数字节长度。
- \(h\)：hypertree 总高度。
- \(d\)：hypertree 层数。
- \(k,a\)：FORS 参数。
- \(w\)：WOTS+ Winternitz 参数。
- \(q\)：每把密钥允许的最大签名次数预算。

参数说明文档特别强调，\(q\) 不应被隐藏处理，而应显式纳入设计目标，因为签名次数预算会显著影响最优参数选择。[cite:57]

### 第三步：建立候选过滤器

对每个候选参数组，自动计算并检查：

- WOTS+ 的 \(len_1\)、\(len_2\)、\(len\)。
- FORS 消息编码长度 \(k\cdot a\)。
- \(H_{msg}\) 输出是否足以切分为消息摘要、树索引与叶子索引。
- 理论签名大小。
- 底层 Poseidon2 调用次数。

这一步的目标是快速剔除“不合法”“明显过大”或“无法满足编码约束”的候选，避免把大量无效点送入后续安全评估和基准测试。[cite:58][cite:57]

### 第四步：做解析安全筛选

对于通过结构过滤的候选参数，使用 SPHINCS+ 规范中的安全评估口径估算组合层安全下界。规范第 9 章讨论了 distinct-function multi-target second-preimage resistance、函数族伪随机性与 interleaved target subset resilience，并在给定参数集时评估安全级别。[cite:58]

建议把候选参数的声称安全值定义为下式：

\[
\text{claimed security} = \min\{\text{SPHINCS+ 组合层下界},\ \text{Poseidon2 实例下界},\ \text{签名预算 } q \text{ 下的下界}\}
\]

只有当这个最小值不低于 128 时，才把该参数组保留为“128-bit 目标候选”。这是数学上的下界判定，不是经验猜测。[cite:58][cite:57][cite:12]

### 第五步：实现评测与 Pareto 筛选

对解析上达标的候选参数，进行以下实现测量：

- keygen 时间。
- sign 时间。
- verify 时间。
- 公钥、私钥、签名字节数。
- 各函数族的 Poseidon2 调用次数。
- 电路约束数，最好分解到 \(F/H/T_\ell/PRF/H_{msg}\) 层面。

参数说明文档强调，不应只看单一指标，而应保留多个维度上的 Pareto 最优点；你的工作还应把“约束数”加入这一前沿分析中。[cite:57][cite:12]

### 第六步：超预算签名退化分析

对最终入围的参数组，还需要评估当签名次数从设计预算 \(q\) 增大到 \(2q\)、\(2^{20}\)、\(2^{30}\) 等水平时，安全下界如何变化。参数说明文档专门讨论了这一点，说明它属于正式参数设计的一部分。[cite:57]

## 如何判断“达到 128 位安全强度”

### 简短结论

是的，需要数学分析，但通常不需要从头重写整个 SPHINCS+ 证明。更准确地说，需要完成“组合层参数评估 + Poseidon2 实例化假设说明 + 签名次数预算分析”三部分工作。[cite:58][cite:59][cite:57]

### 最低判定标准

可以把“达到 128 位安全强度”的最低判定标准写成下面三条同时成立：

1. 按 SPHINCS+ 规范方法计算得到的组合层安全下界不低于 128。[cite:58]
2. 选定的 Poseidon2 参数在公开分析框架下，其碰撞、原像、第二原像与区分安全主张不低于目标值，且接口封装不会破坏域分离。[cite:12]
3. 在给定签名预算 \(q\) 内，安全退化后的最小下界仍不低于 128。[cite:57][cite:58]

### 论文中可接受的表述

若上述三项都完成，建议在论文中使用以下表述方式：

- “该参数集面向 128-bit targeted classical security 设计。”
- “在 SPHINCS+ simple/ROM 假设下完成参数评估。”
- “底层采用 Poseidon2 实例化，并给出接口级安全假设说明。”

若 Poseidon2 实例化分析尚不够完备，则更稳妥的表述是“通过了基于 SPHINCS+ 参数评估框架的 128-bit 目标筛选”，而不是直接写成“已严格证明达到 128 位安全”。[cite:58][cite:59][cite:12]

## 建议的可交付结果

### D1. 接口实例化说明书

一份 3 到 6 页的技术说明，内容包括：

- Poseidon2 参数表。
- 各 SPHINCS+ 接口的输入输出格式。
- ADRS 编码与域分离规则。
- 截断规则与消息映射方法。
- simple 版本的安全语境说明。

这是后续所有参数重搜与安全表述的依据。[cite:58][cite:12]

### D2. 参数搜索脚本

一份可复现实验脚本，输入候选 \((n,h,d,k,a,w,q)\)，输出：

- 结构合法性检查结果。
- 理论签名大小。
- 各模块调用次数。
- 组合层安全估计值。
- 是否通过 128-bit 门槛。

规范和参数说明都强调参数评估应可复现，因此脚本是关键可交付件。[cite:58][cite:57]

### D3. 候选参数总表

至少包含以下列：

| 参数 | 说明 |
|---|---|
| \(n,h,d,k,a,w,q\) | 参数本体 |
| pk/sk/sig 大小 | 规模指标 |
| keygen/sign/verify | 软件性能指标 |
| Poseidon2 调用总数 | 哈希代价指标 |
| 约束数 | 零知识友好指标 |
| 组合层安全下界 | SPHINCS+ 评估值 |
| Poseidon2 实例下界 | 底层安全主张 |
| claimed security | 最终主张 |

该表应同时保留所有达标点与最终 Pareto 前沿点。[cite:57][cite:58][cite:12]

### D4. 最终推荐参数集

建议最终给出三组参数，而不是只给一组：

- 最小签名参数组。
- 最小约束参数组。
- 综合平衡参数组。

这样更符合参数搜索的研究规范，也更便于论文中展示“为什么选择这组参数”。[cite:57]

### D5. 安全表述模板

准备一段可直接进入论文的安全表述模板，例如：

> 本文针对 SPHINCS+ simple 实例化，采用 Poseidon2 构造 \(T_\ell\)、\(F\)、\(H\)、\(PRF\)、\(PRF_{msg}\) 与 \(H_{msg}\) 接口，并依据 SPHINCS+ 既有参数评估框架对候选参数进行筛选。在给定签名预算下，本文选择的参数集面向 128-bit targeted classical security 进行设计与评估。[cite:58][cite:59][cite:57]

## 推荐的执行顺序

1. 先完成接口实例化说明书。没有这一步，后续“达到 128 位”的说法缺乏对象定义。[cite:58]
2. 再实现参数搜索脚本，先做结构合法性检查，再做安全过滤。[cite:58][cite:57]
3. 对通过过滤的候选做软件 benchmark 与约束数统计。[cite:57][cite:12]
4. 输出 Pareto 前沿，并挑选 3 组代表参数。[cite:57]
5. 最后撰写安全表述，不要先写结论再补分析。[cite:58][cite:59]

## 结题判据

当以下条件同时满足时，可以认为本阶段目标完成：

- 已有一份清晰的 Poseidon2-SPHINCS+ simple 接口实例化定义。[cite:58]
- 已有一套可复现的参数搜索与过滤脚本。[cite:57][cite:58]
- 已找到至少一组在解析评估下达到 128-bit 目标的候选参数。[cite:58]
- 已完成性能、大小和约束数测量，并输出 Pareto 前沿结果。[cite:57][cite:12]
- 已形成谨慎、准确、可发表的安全表述文本。[cite:58][cite:59]
